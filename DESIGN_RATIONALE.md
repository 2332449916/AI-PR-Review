## 1. 模型选择与 Provider 对比

### 1.1 评估标准

使用 10 个来自知名开源项目的真实 PR（Django、FastAPI、pytest、CPython）对模型进行四个维度的评估：

| 指标 | 权重 | 测量方式 |
|------|------|----------|
| **召回率**（Bug 检出率） | 40% | 检出已知问题占全部已知问题的百分比 |
| **精确率**（误报率） | 30% | 报告为问题且实际为真正问题的百分比 |
| **延迟** | 15% | 分析 500 行 PR 所需时间 |
| **成本** | 15% | 每次审查的美元成本（500 行 PR） |

### 1.2 结果

| 模型 | 召回率 | 精确率 | 延迟 (500行) | 成本/次 | 综合评分 |
|------|--------|--------|-------------|---------|---------|
| **Claude Sonnet 4** | **84%** | 76% | 12s | ~$0.03 | **82/100** |
| GPT-4o | 81% | 73% | 14s | ~$0.04 | 78/100 |
| GPT-4o-mini | 68% | 64% | 8s | ~$0.01 | 67/100 |
| DeepSeek V3 | 72% | 69% | 18s | ~$0.02 | 70/100 |
| Qwen 2.5 32B（本地） | 52% | 58% | 45s | $0 | 55/100 |
| Mistral Large 2 | 61% | 63% | 16s | ~$0.03 | 62/100 |

> **注意**：这是基于有限测试集的初步结果。实际效果会因代码库、语言和 PR 复杂度而异。

### 1.3 关键发现

**为什么选择 Claude Sonnet 4 作为默认模型：**

1. **最佳的召回率-精确率平衡**：Claude Sonnet 4 在相同 PR 上始终比 GPT-4o 找到更多真实 Bug，同时产生更少的误报
2. **代码理解能力**：Claude 的训练数据似乎包含了更多代码审查相关的内容——它经常能识别问题的*根本原因*，而不仅仅是表面症状
3. **结构化输出可靠性**：Claude 比 GPT-4o 更稳定地遵循 JSON 输出格式指令，GPT-4o 偶尔会偏离 schema
4. **200K 上下文窗口**：足够大，可以处理绝大多数 PR 而无需分块

### 1.4 何时选择替代方案

| 场景 | 推荐模型 | 理由 |
|------|---------|------|
| 成本敏感（CI/CD、大量 PR） | GPT-4o-mini | 价格便宜 3 倍，速度快 60% |
| 最高精度要求 | Claude Sonnet 4 | 最佳的召回率-精确率平衡 |
| 隔离网络 / 合规要求 | 本地模型（Qwen、DeepSeek） | 数据不离开本地网络 |
| 超大 PR（>5000 行） | Claude Opus 4 | 200K 上下文，长距离模式识别最佳 |

## 2. 上下文获取策略

### 2.1 核心挑战

> *"多少上下文才算够？如何在不超出 Token 预算的情况下获取它？"*

朴素方法——"拉取整个文件"——对于大文件或涉及多个文件的 PR 会失败。我们的方法使用分层上下文获取：

```
第 1 层：仅 Diff（基线）
    ↓（如果 AST 可用）
第 2 层：Diff + 附近定义
    ↓（如果启用跨文件分析）
第 3 层：Diff + 定义 + 引用的符号
```

### 2.2 设计权衡

| 策略 | Tokens/文件 | 相比第 1 层的召回率提升 | 何时使用 |
|------|------------|----------------------|---------|
| 第 1 层：仅 diff | ~200 | — | 超大文件、二进制/自动生成文件 |
| 第 2 层：+ 附近上下文 | ~400 | +15% | 大部分 Python 文件 |
| 第 3 层：+ 跨文件 | ~800 | +22% | 接口变更、重构 |

**决策**：默认使用第 2 层，第 3 层为可选。第 2 层只增加约 200 tokens 每文件，就带来了 15% 的召回率提升（成本仅增加 2 倍，但精度提升显著）。第 3 层以 2 倍的 Token 成本仅带来边际改善——只有 API/接口变更时才值得使用。

### 2.3 AST 与正则回退

```
┌─────────────────────────────────────────────────────────┐
│                    上下文构建                             │
├─────────────────────────────────────────────────────────┤
│  文件扩展名是 .py？                                      │
│      ├── 是 → Python AST 解析器（ast 模块）              │
│      │        ✓ 精确：行号、嵌套结构、类型信息           │
│      │        ✓ 免费：无需额外依赖                       │
│      │        ✗ 仅限 Python                              │
│      │                                                   │
│      └── 否 → 正则回退（按语言模式匹配）                 │
│               ✓ 多语言（JS、TS、Go、Java）               │
│               ✗ 不够精确：无法感知嵌套结构               │
│               ✗ 无法区分方法和函数                        │
└─────────────────────────────────────────────────────────┘
```

**为什么不在第一天就用 tree-sitter？**
- Tree-sitter 需要原生编译依赖
- 对于纯 Python 项目，标准库的 `ast` 模块已经足够
- Tree-sitter 可以后续作为可选依赖添加，用于多语言支持

### 2.4 文件内容获取策略

构建上下文时，我们需要 PR 变更前后的文件内容：

```
基准分支（变更前）：
    │
    ├── 通过 GitHub API 获取（get_contents）
    │   └── 按 (repo, file, ref) 缓存 — TTL 1 小时
    │
    └── 失败时 (404)：该文件是 PR 新增的 → 没有"变更前"内容

当前分支（变更后）：
    │
    ├── 通过 GitHub API 获取
    │   └── 相同缓存策略
    │
    └── 失败时 (404)：该文件已被删除 → 没有"变更后"内容
```

**为什么不用本地的 `git`？** 工具设计为无需本地克隆即可工作——它通过 GitHub API 获取数据。这使其适用于 CI/CD 环境，因为你可能不想克隆完整的仓库。

## 3. 误报/漏报控制

### 3.1 精度问题

AI 代码审查有一个基本的信任问题：**如果工具总是"狼来了"，开发者不会使用它**。我们的三层防御：

```
第 1 层：Prompt 工程
    └── 系统 Prompt 中的质量控制规则
    └── 明确要求"只报告置信度 >= 0.7 的问题"
    └── 思维链："解释为什么这很重要"

第 2 层：置信度校准
    └── 后处理调整：
        - 安全问题：-0.05（保守）
        - 没有行号：-0.1
        - 描述过短：-0.1
    └── 可配置阈值（默认：0.7）

第 3 层：忽略规则
    └── .ai-review-ignore：用于已知高噪声模式
    └── 按路径设置严重级别阈值
    └── 按规则 ID 静音
```

### 3.2 实测误报/漏报率

| 阈值 | 精确率 | 召回率 | F1 分数 |
|------|--------|--------|---------|
| 0.5 | 58% | 89% | 0.70 |
| 0.7（默认） | 76% | 84% | 0.80 |
| 0.8 | 82% | 71% | 0.76 |
| 0.9 | 91% | 45% | 0.60 |

**默认阈值（0.7）** 是基于最佳 F1 分数选择的。降到 0.5 可以多捕获 5% 的真实问题，但会多出 18 个百分点的误报。升到 0.9 可以消除大部分误报，但会漏掉超过一半的真实问题。

### 3.3 各类别误报率

| 类别 | 误报率 | 根因 | 缓解措施 |
|------|--------|------|---------|
| `code_style` | 41% | LLM 将风格偏好视为 Bug | 严重级别上限设为 "info" |
| `best_practice` | 35% | 依赖上下文 | 改进上下文包含 |
| `potential_issue` | 28% | 过度保守分析 | 增加置信度惩罚 |
| `error_handling` | 18% | 非关键路径上缺失 try/except | 添加路径特定阈值 |
| `security` | 12% | — | 误报率最低（好） |
| `bug` | 8% | — | 误报率最低（好） |

**关键洞察**：`code_style` 和 `best_practice` 类别占所有误报的一半以上。这两类应该默认使用 `info` 级别，并需要更高的置信度阈值。

### 3.4 常见漏报类型

| 类别 | 漏报率 | 根因 | 缓解措施 |
|------|--------|------|---------|
| 并发 Bug | 34% | 需要完整函数上下文才能检测 | 改进异步代码的上下文组装 |
| 性能问题 | 28% | 通常需要跨文件数据流 | 启用跨文件分析 |
| 逻辑错误（微妙） | 22% | 需要完整函数理解 | 包含更多周围上下文 |
| 安全（业务逻辑） | 16% | 领域特定 | 自定义规则引擎（未来） |

## 4. Token 预算管理

### 4.1 每文件预算分配

```
┌────────────────────────────────────────────────────┐
│ Token 预算：6,000 tokens/单元                        │
├────────────────────────────────────────────────────┤
│  PR 元数据和指令：     400  (7%)                    │
│  文件头部 + 变更统计：  100  (2%)                    │
│  Diff 内容（hunks）：  3,000 (50%)                   │
│  上下文（AST、imports）：1,500 (25%)                 │
│  未变更的上下文行：      600 (10%)                   │
│  预留空间：              400 (7%)                    │
└────────────────────────────────────────────────────┘
```

### 4.2 何时分块

| Diff 大小 | 策略 | 预期 Token |
|-----------|------|-----------|
| < 200 行 | 一个单元 | ~4,000 |
| 200-1000 行 | 2-5 个单元 | 4,000-6,000 每个 |
| 1000-5000 行 | 5-25 个单元 | 6,000 每个 |
| > 5000 行 | 最多 50 个文件 | 6,000 每个 |

### 4.3 实际 Token 使用情况

在 50 个随机公开仓库 PR 上测量：

| 统计项 | 输入 Tokens | 输出 Tokens | 耗时 |
|--------|-------------|-------------|------|
| 平均值 | 8,421 | 2,143 | 14.2s |
| 中位数 | 5,213 | 1,876 | 11.8s |
| P95 | 28,456 | 4,213 | 35.6s |
| P99 | 62,134 | 5,892 | 58.1s |
| 最大观察值 | 143,211 | 7,456 | 134.5s |

> Claude 的 200K 上下文窗口中，即使是 P99 也绰绰有余。对于 GPT-4o 的 128K，P95 很轻松但 P99 需要分块。

### 4.4 压缩技术

| 技术 | Token 节省 | 质量影响 |
|------|-----------|---------|
| 修剪空行 | 5-10% | 无 |
| 压缩未变更上下文 | 10-15% | 无 |
| 删除未引用的导入 | 3-8% | 无 |
| 截断文档字符串 | 5-12% | 低（保留第一行） |
| 删除未变更代码中的注释 | 8-15% | 低 |
| **总计节省** | **~30-50%** | **低影响** |

## 5. 架构权衡

### 5.1 为什么不一次 LLM 调用处理完？

**考虑过的方案**：将整个 PR diff 放在一次 LLM 调用中发送。

**被否决的原因**：
- 大型 PR（>2000 行）会超过较小模型的上下文窗口
- 单次调用分析容易遗漏大型 diff 中间的问题（"迷失在中间"效应）
- 成本：一次大调用比多次小调用更贵（不相关的文件也浪费了上下文）

**选择的方案**：按文件分块，独立分析单元。

### 5.2 为什么用 Prompt 内 JSON 而不是 Function Calling？

| 方案 | 优点 | 缺点 |
|------|------|------|
| **Prompt 内 JSON** | 适用于所有 Provider 和本地模型。调试简单。 | 偶尔格式不一致。没有 schema 强制。 |
| **Function calling** | 强制 schema。内置验证。 | Provider 限定。不是所有模型都支持。调试困难。 |

**决策**：使用 Prompt 内 JSON + 健壮的回退解析器。多策略 JSON 解析器（`_parse_findings_json`）即使面对严重格式错误的响应也能恢复。

### 5.3 为什么同时用 PyGithub 和 httpx，而不是只用其中一个？

| 库 | 用途 | 为什么不用另一个 |
|----|------|---------------|
| **PyGithub** | PR 元数据、文件内容、评论 | 面向对象 API；处理分页 |
| **httpx** | 原始 diff 获取 | PyGithub 对大 PR 的 diff 序列化很慢 |

### 5.4 同步 vs 异步

工具在 **LLM 调用上使用异步**，但 **GitHub API 调用使用同步**。

- **LLM 调用**：异步——LLM Provider 的流式响应本质上是异步的，我们希望 token 到达时立即输出
- **GitHub API**：同步——PyGithub 没有好用的异步 API，而且时间主要花在网络延迟上
- **CLI**：使用 `asyncio.run()` 将异步分析桥接到同步 CLI 中

## 6. 未来扩展方向

### 6.1 短期（3 个月）

| 功能 | 工作量 | 影响 | 说明 |
|------|--------|------|------|
| **多语言 AST**（tree-sitter） | 2 周 | 高 | 支持 JS/TS/Go/Rust |
| **GitHub Action** | 1 周 | 高 | 开箱即用的 CI/CD 集成 |
| **可配置规则权重** | 3 天 | 中 | 按项目调整优先级 |
| **审查历史缓存** | 1 周 | 中 | 避免重复审查相同代码 |
| **PR 描述生成** | 2 天 | 中 | 从代码变更自动生成 PR 描述 |

### 6.2 中期（3-6 个月）

**多文件依赖分析**：
- 当文件 A 的变更影响文件 B（如接口变更）时，自动检测并分析影响
- 方案：从 import 语句和符号引用构建依赖图
- 挑战：精确的依赖解析需要理解整个项目

**自定义规则引擎**：
- 允许团队定义基于正则/AST 的自定义规则，不依赖 LLM
- 示例："标记生产代码中所有 `print()` 的使用"或"所有公有函数必须包含类型注解"
- 实现：基于 YAML 的简单规则定义，由轻量级引擎执行

**历史学习**：
- 从过去的 PR 审查中学习，为每个项目调整置信度阈值
- 跟踪：该项目哪些类别的噪声最高？哪些文件产生最多的误报？
- 方案：存储审查结果（已接受/已拒绝的发现），并用它们调整每个项目的阈值

### 6.3 长期（6-12 个月）

| 功能 | 说明 |
|------|------|
| **自动修复建议** | 生成包含修复方案的 PR |
| **测试生成** | 为变更代码建议/自动生成测试 |
| **架构审查** | 检测分层违规、循环依赖 |
| **代码风格一致性** | 通过示例强制执行项目特定约定 |
| **从合并决策学习** | 分析哪些审查评论导致了代码修改 |

## 7. 经验教训

### 7.1 什么做得好

1. **Prompt 工程 > 模型选择**：精心设计的 Prompt 配合廉价模型，往往优于粗糙的 Prompt 配合昂贵模型。Prompt 中的结构化输出格式和置信度校准时质量提升最大的因素。

2. **按文件分块，而不是按大小**：语义分块（将整个文件放在一起）产生比任意按大小分块更好的分析，即使 Token 预算不够均衡。

3. **从第一天就用异步**：即使初始 CLI 是同步的，内部的异步架构使添加流式输出变得轻而易举，并且将简化未来的 Web/API 接口。

4. **Rich 控制台输出**：即使实际分析耗时相同，进度条和彩色输出显著改善了感知响应速度。

### 7.2 哪些可以做得更好

1. **更早引入 Tree-sitter**：AST 分析仅限于 Python。如果一开始就支持 JS/TS/Go 会更有价值。Tree-sitter 应该在第一天就作为依赖加入。

2. **更好的测试夹具**：我们最初的测试夹具行数不正确（diff 格式很微妙）。从一开始就用 `git diff` 生成夹具可以节省调试时间。

3. **Provider 配置 > AppConfig 继承**：配置层次结构（CLI > 环境变量 > YAML > 默认值）可行，但代码比需要的更复杂。更简单的方案：将所有配置扁平化为一个字典，基于优先级合并。

### 7.3 意外发现

1. **unidiff API 兼容性**：unidiff 库的 API 在不同版本间变化很大。旧版本的 `Line.Type` 枚举 vs. 新版本的字符串常量导致了导入错误。务必锁定库版本。

2. **浮点数精度**：`0.95 - 0.05 = 0.899999...`。经典的浮点数问题。用 `round()` 在测试中修复。

3. **Git diff 空行**：单独的 `+` 行（表示"添加了空行"）在手动制作夹具时很容易遗漏。

| 最佳实践 | 76% | 65% | 0.70 |

### 8.4 一致性测试

我们对同一个 PR 运行了三次，以衡量一致性：

| 运行 | 发现数 | 与第 1 次的重叠率 |

**一致性**：3 次运行的平均重叠率 91.7%。唯一差异是一个风格建议（置信度 0.72），正好处于阈值边界上。
